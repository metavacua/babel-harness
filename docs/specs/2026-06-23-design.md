# Claude Code × Pi Agent Integration

**Date:** 2026-06-23
**Status:** Approved for implementation

## Goal

Claude Code operates as the primary orchestrator and full-stack operator of a local AI coding pipeline. It configures, repairs, and delegates tasks to the Pi agent harness, which routes work between OpenRouter free-tier models (default) and local Ollama models (offline fallback). Claude Code handles the entire setup — no manual configuration steps remain for the user.

The long-term goal is for the local provider slot (currently Ollama) to be served by custom models constructed and compiled via the `larql` toolchain from `metavacua/larql-to-sparql`. In the larql model, a transformer is decompiled into a **vindex** (a queryable discrete geodesic graph stored as `.bin` files), modified through LQL queries, compiled back via `COMPILE CURRENT INTO VINDEX`, and served via `larql-server`. The harness architecture is designed to accommodate this third provider path without requiring a redesign.

## Problem Stack

Four problems must be resolved in order:

1. **Ollama service broken** — the `ollama` system user referenced in `/etc/systemd/system/ollama.service` was never created; the service exits immediately (systemd code 217/USER).
2. **Model tool-calling misconfigured** — `phi3:mini` does not reliably support Pi's tool-call format; `deepseek-coder:6.7b` can work but requires a correct Modelfile template. Without it, Ollama rejects or mangles tool calls, producing the "not configured for coding tool usage" errors.
3. **Pi OpenRouter↔Ollama handoff incomplete** — Pi's `~/.pi/agent/settings.json` lists both provider families but lacks a routing policy that automatically falls back to Ollama when OpenRouter is unreachable.
4. **No resource envelope** — running Claude Code + Ollama + Pi simultaneously on 6.3GB RAM with no swap is unsafe without explicit memory ceilings. A quantized 7B model needs ~4–5GB alone.

## Architecture

```
Claude Code (orchestrator)
    │
    ├─ Read/Edit ──► ~/.pi/agent/settings.json
    ├─ Read/Edit ──► /etc/systemd/system/ollama.service
    ├─ Read/Edit ──► Ollama Modelfiles
    │
    └─ Bash ──► pi-harness <task>
                    │
                    ├─ [check] OpenRouter reachable?
                    │     yes ──► pi --provider openrouter --model openrouter/free --print --mode json
                    │     no  ──► [check] Ollama healthy?
                    │                 yes ──► pi --provider ollama --model qwen2.5-coder:7b --print --mode json
                    │                 no  ──► exit 1 with clear error
                    │
                    └─ [enforce] cgroup memory ceiling (5GB total)

Future provider slot (deferred — replaces or supplements Ollama):
    └─ Bash ──► pi-harness <task> --provider larql
                    │
                    └─ larql-server (serving a compiled .vindex)
                           ├─ extract: larql extract <hf-model> → .vindex
                           ├─ query/edit: LQL (DESCRIBE / INSERT / COMPILE)
                           └─ serve: larql-server --vindex <path> --port <n>
```

## Solution Layers

### Layer 1 — Ollama service repair

Change `User=ollama` to `User=metavacua` in `/etc/systemd/system/ollama.service`. This requires no new system user and keeps the model store at `~/.ollama/` where it already lives. After editing: `sudo systemctl daemon-reload && sudo systemctl restart ollama`.

Claude Code performs this repair autonomously when it detects the service is not running.

### Layer 2 — Model tool-calling fix

Pull `qwen2.5-coder:7b` as the primary local coding model — it has first-class tool-call support in Ollama and fits within ~4.5GB quantized. Create an Ollama Modelfile that extends it with an explicit tool-use template if needed.

`phi3:mini` is retained as a lightweight fallback for non-tool queries only (summarisation, quick lookups). `deepseek-coder:6.7b` is deprecated in Pi's `enabledModels` unless a working Modelfile can be validated.

Update Pi's `settings.json` `enabledModels` to reflect the new model set.

### Layer 3 — Pi handoff completion

**Default provider:** `openrouter`, model `openrouter/free`
**Fallback provider:** `ollama`, model `qwen2.5-coder:7b` (only when OpenRouter unreachable)

The routing policy lives in `pi-harness` (Layer 4) rather than inside Pi's own config, so Pi's settings remain a simple list of enabled models. Pi's `defaultProvider` stays `openrouter` and `defaultModel` stays `openrouter/free`, matching the existing config.

### Layer 4 — `pi-harness` script

**Location:** `~/.local/bin/pi-harness`
**Pattern:** mirrors `larql-probe` cgroup harness

Responsibilities:
- **Reachability check**: `curl -sf --max-time 3 https://openrouter.ai/api/v1/models > /dev/null`; use the exit code to select provider
- **Ollama health check** (fallback path only): verify Ollama is serving (`ollama ps` or `curl localhost:11434`); if not, attempt `sudo systemctl restart ollama` (user has NOPASSWD sudo); if still down, exit with error
- **cgroup enforcement**: set memory limit (5GB) and CPU cap on the Ollama + Pi process tree using cgroup v1 (`/sys/fs/cgroup/memory/`) to prevent OOM (kernel 6.6 on this machine uses v1 hierarchy)
- **Invocation**: call Pi with `--print --mode json` and the selected provider/model flags
- **Exit**: return Pi's JSON stdout and Pi's exit code to Claude Code unchanged

Interface:
```bash
pi-harness "refactor the auth module to use dependency injection"
pi-harness --model phi3:mini "summarize this file"   # force a specific model
pi-harness --status                                  # report Ollama + OpenRouter health
pi-harness --repair                                  # attempt to fix Ollama service
```

### Layer 5 — Claude Code management interface

No additional tooling. Claude Code uses its existing tools:

- **Bash**: invoke `pi-harness`, run `systemctl`, run `ollama pull/ps/show`, check connectivity
- **Read/Edit**: manage `~/.pi/agent/settings.json`, Modelfiles, `ollama.service`
- **Delegation heuristic** (encoded in project CLAUDE.md): delegate to `pi-harness` for mechanical, well-defined, in-repo coding tasks; handle inline for architecture decisions, novel reasoning, or tasks requiring conversation history

## Resource Budget

| Component | Estimated RAM |
|-----------|--------------|
| Claude Code process | ~300MB |
| Ollama server (idle) | ~200MB |
| qwen2.5-coder:7b (quantized, loaded) | ~4.5GB |
| Pi process | ~150MB |
| **Total (Ollama active)** | **~5.1GB** |
| **Total (OpenRouter path, Ollama idle)** | **~0.7GB** |

Cgroup ceiling: 5GB. On the OpenRouter path Ollama is not loaded into memory, so the machine operates comfortably. On the Ollama fallback path, the ceiling prevents runaway usage but leaves ~1GB headroom for the OS and other processes.

## Out of Scope

- Pi delegating back to Claude Code (future goal B) — deferred
- MCP server wrapping Pi — deferred until basic delegation is proven
- Automatic model selection based on task complexity — deferred; manual `--model` override flag covers the interim need

## Future: larql Provider Path

The `pi-harness` `--provider` flag is intentionally generic. Once `larql-server` is stable in `metavacua/larql-to-sparql`, the local fallback can be extended or replaced:

1. **Extract**: `larql extract <hf-model-id> --tier inference` → `.vindex` directory
2. **Customise** (optional): LQL queries to insert domain-specific edges, then `COMPILE CURRENT INTO VINDEX`
3. **Serve**: `larql-server --vindex <path> --port 11435` (OpenAI-compatible endpoint)
4. **Wire**: add `larql` as a Pi provider pointing at `localhost:11435`; update `pi-harness` fallback chain to prefer `larql` over Ollama when the server is healthy

Key repos: `metavacua/larql-to-sparql` (toolchain, vindex format spec at `docs/specs/vindex-format-spec.md`), `metavacua/GeodesicLangModel` (theoretical framework). `metavacua/mlc-llm` fork is early-stage placeholder for MLC compilation integration — not yet actionable.
